Cloud Foundry 扩大了对 Kubernetes TechCrunch 的支持

原文：https://web.archive.org/web/https://techcrunch.com/2018/10/10/cloud-foundry-expands-its-support-for-kubernetes/

不久前， Cloud Foundry Foundation 完全是关于 Cloud Foundry 的，这是一个开源平台即服务(PaaS)项目，现在大多数财富 500 强企业都在使用它。这个项目是云铸造应用运行时。一年前，该基金会还宣布了 Cloud Foundry Container Runtime，帮助企业并行运行应用平台及其基于容器的应用。此外，Cloud Foundry 也长期以来一直是 BOSH 背后的力量，BOSH 是一种用于构建、部署和管理云应用程序的工具。

Container Runtime a year go 的加入似乎有点混淆了组织的使命，但是现在尘埃落定，这里的意图开始变得更加清晰。正如 Cloud Foundry CTO Chip Childers 告诉我的，企业使用容器运行时的主要目的是运行他们从供应商那里获得的预打包应用程序。“容器运行时——或者实际上是 Kubernetes 的任何部署——当与应用运行时一起使用或结合使用时，人们主要是在那里登陆由独立软件供应商提供的打包软件，”他告诉我。“容器是新的光盘。你只是想把它放在一个好的编排平台上。”

因为应用程序运行时在 Kubernetes 出现之前就已经发布了，所以 Cloud Foundry 项目构建了自己的容器服务，称为 Diego。

今天，Cloud Foundry foundation 推出了两个与 Kubernetes 相关的新项目，将两者之间的集成提升到了一个新的水平。第一个是由 IBM 发起的 Eirini 项目，Suse 和 SAP 目前也在从事这项工作。这个项目已经酝酿了很长时间，也是社区期待已久的事情。在部署为应用程序运行时编写的应用程序时，它基本上允许开发人员在使用现有的 Diego orchestrator 和 Kubernetes 之间进行选择。这对 Cloud Foundry 来说是一件大事。

“Eirini 所做的是利用 Cloud Foundry 应用运行时——与[Cloud Foundry]品牌紧密相关的核心 PaaS 体验，并允许用 Kubernetes 替换底层 Diego 调度程序，作为它所覆盖的用例的一个选项，”Childers 解释道。他补充说，仍然有一些用例迭戈集装箱管理系统比 Kubernetes 更适合。其中之一是更好的 Windows 支持——这对使用 Cloud Foundry 的企业来说相当重要。Childers 还指出，Kubernetes 的多租户担保没有 Diego 的严格。

第二个新项目是 CF 集装箱化，最初由 Suse 开发。顾名思义，CF 容器化基本上允许您打包核心 Cloud Foundry 应用程序运行时，并借助 BOSH 部署工具将其部署在 Kubernetes 集群中。这几乎就是 Suse 已经用来发布其 Cloud Foundry 发行版的东西。

很明显，Kubernetes 正在成为 Cloud Foundry PaaS 服务的一部分，在不久的将来，开发者将使用它来部署他们为之编写的应用程序。乍一看，这种对 Kubernetes 的关注可能会使 Cloud Foundry 变得多余，但值得记住的是，在其核心，Cloud Foundry 应用程序运行时不是关于基础设施，而是关于旨在管理整个应用程序开发生命周期的开发人员体验和方法。如果 Kubernetes 可以用来帮助管理基础设施，那么 Cloud Foundry 项目也可以专注于它最擅长的领域。
