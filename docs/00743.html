地平线上有一片乌云。亚马逊等云基础设施提供商的行为威胁着开源的生存能力。我首先在TechCrunch上的一篇 前作 中写了这个问题。2018年，谢天谢地，几位领导人已经动员起来(在争议中)提出了解决问题的多种方案。这是上个月发生的事情。

问题是

转到 亚马逊网络服务 (AWS)并将鼠标悬停在顶部的产品菜单上。你会看到许多开源项目，这些项目不是亚马逊创建的，而是作为服务运行的。这些每年为亚马逊提供数十亿美元的收入。明确一点，这不违法。但它不利于可持续的开源社区，尤其是商业开源创新。

两种解决方案

2018年初，我召集了二十几家大规模开源公司的创作者、首席执行官或总法律顾问，以及受人尊敬的开源律师 希瑟·米克 ，来讨论该怎么办。

我们希望定义一个许可证，防止云基础设施提供商将某些软件作为商业服务运行，同时使该软件对其他所有人有效开源，即所有人都不将该软件作为商业服务运行。

对于我们的第一个提案，Commons Clause，我们采取了最直接的方法:我们构建了一个条款，它可以添加到任何自由开源许可证中，防止被许可方“出售”软件——其中“出售”包括将其作为商业服务运行。(当然，销售使用Commons Clause软件制作的其他软件是允许的。)应用Commons子句将项目从开放源代码转换为源代码可用。

我们也喜欢由另一个参与者MongoDB率先提出的建议，称为 【服务器端公共许可证(SSPL) 。SSPL并未禁止将软件作为服务运行，而是要求您开源所有用于将软件作为服务提供的程序，包括但不限于管理软件、用户界面、应用程序界面、自动化软件、监控软件、备份软件、存储软件和托管软件，所有这些都是为了让用户能够运行服务的实例。这就是所谓的“左版权”

这两个许可证是完全相同问题的两种不同解决方案。 希瑟·米克 写了这两个解决方案，由反馈支持 窝 。

最初的喧嚣和指责是这些努力试图“欺骗”社区，幸运的是，这些努力让位于来自开源社区的理解和承认，这里有一个真正的问题需要解决，是时候让开源社区得到真正的 了，是时候让网络巨头们为他们所依赖的开源软件 付出公平的代价了。

十月份，Apache Software Foundation (ASF)的一名董事会成员找到我，建议我们合作创建一个现代的开源许可证来解决行业需求。

向MongoDB致敬

更值得称赞的是，MongoDB明确声明他们将使用SSPL，同时将SSPL提交给一个名为开源倡议(OSI)的组织作为开源许可证进行认可，但没有等待OSI的认可就开始在SSPL下发布软件。

OSI以某种方式将自己标榜为“决定”一个许可是否开源的团体，它有一个短视的习惯，即争论什么是开源，什么不是。随着SSPL提交给OSI，MongoDB已经把球放在OSI的球场上，要么站出来帮助解决一个行业问题，要么把他们的头放回沙子里。

事实上，MongoDB帮了OSI一个大忙。MongoDB已经解决了这个问题，并把一个完全可用的开源许可证放在银盘上交给了OSI。

开源香肠

OSI关于SSPL的辩论的 公共档案 有时内容丰富，有时有趣，近乎滑稽。在MongoDB的 原始提交 之后，成员中出现了一片反对声，认为SSPL不是开源许可证，随后是一些+1的成员John Cowan提醒 组织说 仅仅因为OSI不认可一个许可证是开源的，并不意味着它不是开源的

据我所知(相当远)，OSI不这么做。他们从未公开说过“License X不是开源的。”各种邮件列表上的人都这样做了，但OSI本身没有。他们当然不会说“任何不在我们的OSI Certified列表上的许可证都不是开源的”，因为那是错误的。很容易编写一个显然是开源的许可证，但出于各种原因，OSI永远不会认证它。

Eliot Horowitz(MongoDB的首席技术官和联合创始人) 对问题、评论和异议做出了中肯的回应 ，并总结道:

简而言之，我们认为在当今世界，链接已经被提供程序作为服务和通过网络连接程序作为程序组合的主要形式所取代。目前尚不清楚现有的左版权许可证是否明确适用于这种形式的程序组合，我们希望SSPL成为开发者解决这种不确定性的一种选择。

自MongoDB提交新文件后，OSI有60天时间做出选择:

  1. 清醒过来，意识到SSPL，也许经过一些编辑，确实是一个开源许可，或者

这里的“Wonk”是最好的意思。

重要的是，MongoDB正在不顾一切地使用SSPL。如果MongoDB要等到OSI的决定，或者如果OSI更相关，我们可能会屏息以待，听听OSI是否会支持SSPL作为开源许可证。

就目前情况来看，OSI的决定对OSI本身比对行业更重要。它标志着OSI是否希望保持相关性并帮助解决行业问题，或者它是否变得过于短视而无用。由于担心后者，我们向其他组织寻求领导，并与Apache Software Foundation (ASF)合作，他们希望创建一个现代的开源许可证来解决行业需求。

OSI应该意识到，如果他们认为SSPL是开源的，那当然很好，但这并不重要。再次用约翰·考恩 的话说 ，仅仅因为OSI没有认可一个许可为开源，并不意味着它不是开源的。虽然我们非常尊重行业协会的几乎所有成员以及他们在各自领域所做的工作，但要尊重任何自称是“决定”某个许可是否开源的团体的目的和过程变得越来越困难——这是武断和过时的。

正误表

我热切地敦促业界解决这个问题，在早先的一篇文章中，我曾说过“如果一个人为了自己的利益，把别人构建的开源软件原封不动地作为商业服务提供”(就像云基础设施提供商所做的那样)，这“不符合开源的精神”。这是一种夸大，因此，坦率地说，是不正确的。开源政策专家指出了这一点。我显然不介意打破他们的牢笼，但我应该避免对“精神实质”发表声明，以免转移我的主要论点。

结论

云基础设施提供商的行为对开源构成了生存威胁。云基础设施提供商并不邪恶。目前的开源许可证允许他们逐字逐句地使用开源软件，并将其作为商业服务提供，而不会回馈给开源项目或他们的商业牧羊人。问题是开发者没有开源许可替代方案来阻止云基础设施提供商这样做。开源标准组织应该有所帮助，而不是碍手碍脚。我们必须确保开源软件的作者不仅能够生存，而且能够茁壮成长。如果这意味着对云基础设施提供商采取更强硬的立场，那么作者应该有许可证来允许这样做。开源社区应该将此作为当务之急。
