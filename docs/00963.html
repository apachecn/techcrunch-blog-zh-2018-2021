虽然无服务器通常被认为是降低成本和按需大规模扩展的一种方式，但有一个非常令人信服的理由来采用无服务器优先的方法:这是随着时间的推移实现最大开发速度的最佳方式。正确实现并不容易，当然也不是万灵药，但是，如果做得好，它为最大化开发速度铺平了一条非凡的道路，正因为如此，无服务器是当今创始人和投资者中最不被炒作、讨论最少的技术运动。

无服务器的案例从一个简单的前提开始:如果在一个给定的市场中最快的创业公司将会胜出，那么最重要的事情是随着时间的推移保持或提高开发速度。这听起来可能很明显，但是非常非常少的创业公司将保持或提高开发速度作为明确的目标。

“开发速度”，具体来说，意味着你可以向客户交付额外单位价值的速度。当然，可以通过向现有客户提供更多价值，或者通过向新客户提供现有价值(即现有功能)来增加客户价值。

对于许多科技初创公司来说，尤其是在 B2B 领域，这两者都受到开发吞吐量的限制(前者是显而易见的原因，后者是因为新客户的入职往往受到必须由工程师构建的入职自动化的限制)。无服务器到底是什么意思？这有点用词不当。正如云计算并不意味着数据中心消失在太空中——它意味着这些数据中心正在由其他人运行，服务器可以按需供应并按小时付费— 无服务器并不意味着没有任何服务器。

总会有服务器在某个地方。从广义上讲，无服务器意味着您不需要负责这些服务器的所有配置和管理。无服务器的一个很好的定义是按使用付费计算，正常运行时间不受开发者控制。零使用，就有零成本。如果服务中断，您没有责任恢复服务。AWS 在 2014 年通过一个名为 AWS Lambda 的“无服务器计算”平台启动了无服务器运动。

像 AWS 的 EC2 产品这样的“普通”云服务器必须提前配置，并且无论是否使用都按小时计费，而 AWS Lambda 是即时按需配置的，并且只对每个请求计费。Lambda 惊人地便宜:每个请求 0.0000002 美元，每千兆字节秒计算 0.00001667 美元。虽然用户在 EC2 上遇到容量限制时必须增加服务器规模，但 Lambda 将或多或少地无限扩展以适应负载——无需任何手动干预。而且，如果 EC2 实例宕机，开发人员负责诊断问题并让它重新上线，而如果一个 Lambda 死亡，另一个 Lambda 可以取代它的位置。

尽管从成本和容量的角度来看，Lambda 以及类似的服务如 Azure Functions 或 T2 Google Cloud Functions 非常有吸引力，但事实是，省钱和为规模做准备并不是创业公司采用特定技术的好理由。很少有初创公司会因为在服务器上花费过多资金或无法满足客户需求而失败——事实上，针对这些事情中的任何一个进行优化都是一种过早扩展的形式，而在一个或多个维度(招聘、营销、销售、产品功能，甚至层级/头衔)上过早扩展是绝大多数初创公司死亡的主要原因。换句话说，过早地优化成本、规模或正常运行时间是一种反模式。

当人们谈论无服务器的方法时，他们不仅仅是指将运行在服务器上的代码分割成 Lambda 函数，以实现更低的成本和更容易的扩展。一个合适的无服务器架构是一种完全不同的构建现代软件应用程序的方式——一种被称为无服务器、全服务的方法。

它始于对现成平台的积极采用，即托管服务，如 AWS Cognito 或 Auth0 (用户认证—注册并登录—即服务) AWS 步骤功能或 Azure 逻辑应用(工作流-编排-即服务) AWS AppSync (GraphQL 后端-即服务)，甚至更熟悉的服务

Lambda 类产品以服务的形式提供功能，而托管服务以服务的形式提供功能。换句话说，区别在于你为无服务器计算编写和维护代码(例如函数),而提供商为托管服务编写和维护代码。借助托管服务，该平台提供了管理其背后的运营复杂性的功能和。

通过采用托管服务，应用程序的绝大部分“商品”功能(身份验证、文件存储、API 网关等)由云提供商的各种现成平台处理，这些平台由您自己的一层薄薄的“粘合”代码缝合在一起。粘合代码——以及使您的应用程序独一无二的其余业务逻辑——运行在超廉价、无限可扩展的 Lambda(或等效)基础设施上，从而完全消除了对服务器的需求。像我们这样的小型工程团队正在使用它来构建非常强大、易于维护的应用程序，随着应用程序变得越来越复杂，这种架构会产生前所未有的、可持续的开发速度。

采用无服务器、全方位服务的理念是有代价的。构建一个完全无服务器的应用程序需要对短期开发速度进行巨大的打击，因为构建一个“服务”通常比使用一个现成的 AWS 要快得多。当开发者考虑像 Stripe 这样的服务时，“构建还是购买”甚至不是一个问题——使用 Stripe 的支付服务无疑比自己构建支付服务更快。更准确地说，理解 Stripe 的支付模式比理解和建立一个专有的支付模式更快——这既是支付领域复杂性的证明，也是 Stripe 开发的直观服务的证明。

但是对于处理身份验证(Cognito 或 Auth0)或工作流编排(AWS Step 函数或 Azure Logic 应用程序)的开发人员来说，理解和实现服务提供商的模型通常比在应用程序的代码库中实现功能(无论是从头编写还是使用开源库)要慢。通过选择使用托管服务，开发人员有意选择在短期内放慢速度——这对初创公司来说是一颗难以下咽的药丸。可以理解的是，许多人现在选择加快速度，推出自己的产品。

这种方法的问题又回到了软件开发中的一个古老公理:“代码不是资产——代码是债务。”代码要求在会计等式的两边都有一个条目。它是一种资产，使公司能够向客户交付价值，但它也需要维护，必须考虑并随着时间的推移进行分配。同等条件下，初创公司希望尽可能最小的代码库(当然，前提是开发人员不要走得太远，写出聪明但不可读的代码)。更少的代码意味着需要维护的表面积更少，也意味着新工程师在加速阶段需要掌握的表面积更少。

这就是使用托管服务的魔力所在。初创公司将提供商的代码作为一种资产进行有益的使用，而不会在他们的“技术资产负债表”上持有代码债务。相反，代码位于供应商的资产负债表上，供应商的工程师负责维护、改进和记录代码。换句话说，初创公司免费获得自我维护、自我改进、和自我记录的代码——相当于雇佣一流的工程团队致力于代码库的非核心部分。或者更准确地说，以可预测的每次使用成本。与使用诸如 Cognito 或 Auth0 这样的托管服务形成对比。在第一天，也许它不具备初创公司愿望清单上的所有功能。不同之处在于，提供商有一个工程师和产品经理团队，他们的唯一任务是日复一日地改进这项服务。他们激动人心的核心产品是另一家公司的红发继子。

如果一家初创公司的工程团队有一个统一的原则，那就是尽可能少写代码——负责最少的非核心服务。通过采用这种理念，一家初创公司可以构建一个平台，该平台可以以极其可预测的纯可变成本处理数十亿笔交易，几乎没有 devops 监管。

如此懒惰需要惊人的自律。擅长管理无服务器代码库和无服务器基础设施是很重要的。这意味着围绕测试和自动化建立广泛的实践，这意味着更大的前期投资。与托管服务集成会非常痛苦，需要花费数天时间来试图理解所有的差距、问题和边缘情况。实现专有解决方案的诱惑可能是难以置信的，尤其是当这意味着一个故事可以在几分钟或几小时内完成，而不是几天或更长时间。

这意味着当一个服务只能满足开发人员 80%的需求时，需要编写不可靠的解决方案。随着缺失的 20%的功能被释放，这意味着重构代码来移除变通方法，即使它工作得很好，并且改变它没有短期的好处。大量的早期投资意味着无服务器/托管服务优先的方法并不适合每个初创公司。要问的最重要的问题是，我们需要在多长时间内保持快速？如果答案是几天或几周，就像许多非常早期的创业公司一样，这可能不是正确的方法。

但是，如果速度优化的时间尺度已经从天或周转变为月或年，那么就有必要仔细研究一下无服务器化。

招募优秀的工程师非常困难——而且只会越来越困难。这是一个巨大的竞争优势，让那些工程师去构建差异化的业务功能，而你的竞争对手去构建商品化的、无差别的服务，然后在未来几年内继续维护这些服务。当然，在某些情况下，无服务器毫无意义，但这些情况正在快速消失(例如，Lambda 的 5 分钟超时最近增加了两倍，达到 15 分钟)，锁定或延迟等原因通常是毫无意义的或成为过去。

最终，软件创业公司的工作——因此也是创始人的工作——是提供超出竞争能力的客户价值。这项工作归结为最大化开发速度，反过来，归结为尽可能降低复杂性。可能每一个代码库，因此每一个创业公司，都注定要成为“一个大泥球”——这个词是在 1997 年的一篇论文中创造的，用来描述每个软件项目似乎最终注定要成为的“随意构造、蔓延、松散、管道胶带和打包线、意大利面条式代码丛林”。

总有一天，复杂性将增长超过临界点，开发速度将不可逆转地下降，因此创始人的最终工作是尽可能延长这一天。做到这一点的最好方法是将你的泥巴球保持在尽可能小的尺寸——无服务器是迄今为止开发的最强大的工具。
