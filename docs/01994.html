Redis 实验室再次改变其开源许可

原文：https://web.archive.org/web/https://techcrunch.com/2019/02/21/redis-labs-changes-its-open-source-license-again/

刚刚结束在的最新一轮融资的 Redis 实验室，今天宣布改变其 Redis 模块的许可方式。这听起来可能没什么大不了的，但是在开源项目的世界里，许可目前是一个大问题。这是因为像 Redis、MongoDB、Confluent 和其他组织最近引入了新的许可证，使他们的竞争对手更难在不对社区做出贡献的情况下获取他们的产品并将其作为更名服务出售(这些公司中的大多数都直接指出 AWS 是罪魁祸首)。

“一些云提供商反复利用成功的开源项目，却没有对他们的社区做出重大贡献，”Redis 实验室团队今天写道。“他们将不是他们开发的软件重新包装成有竞争力的专有服务产品，并利用他们的商业杠杆从这些开源项目中获得可观的收入。”

这些新许可证的目的是要制止这种情况。

这并不是 Redis 实验室第一次改变其 Redis 模块的许可方式(我在这里强调“Redis 模块”部分，因为这只是关于 Redis 实验室的模块，与 Redis 数据库项目本身的许可方式没有任何关系)。早在 2018 年，Redis Labs 就将其许可证从 AGPL 更改为使用 Commons 条款修改的 Apache 2。“公共条款”是将商业限制置于许可证之上的部分。

几天前，当我们谈到该公司的资金时，Redis Labs 的联合创始人兼首席执行官 Ofer Bengal 告诉我，这引起了很大的轰动。

“当我们推出这种新的许可证时，有许多不同的观点，”他承认。“有些人对此表示谴责。但是在最初的喧嚣平静下来之后，特别是在其他一些公司提出了类似的概念之后，社区现在明白开源的最初概念必须得到修正，因为它不再适合现代时代，在现代时代，云公司利用他们的垄断权力采用任何成功的开源项目，而不对其做出任何贡献。”

不过，该公司现在表示，代码的许可方式造成了一点混乱，因为一些用户认为他们只受 Apache 2 许可条款的约束。公地条款中的一些术语也不太清楚(例如，包括“实质性”的含义)。

所以今天，Redis 实验室推出了 Redis Source Available 许可证。该许可证也仅适用于 Redis 实验室创建的某些 Redis 模块。用户仍然可以获得代码，修改代码并将其集成到他们的应用程序中，但应用程序不能是数据库产品、缓存引擎、流处理引擎、搜索引擎、索引引擎或 ML/DL/AI 服务引擎。

根据定义，开源许可不能有限制。这个新的许可确实如此，所以从技术上讲，它不是一个开源许可。该公司认为，在实践中，它与其他许可的开源许可证非常相似，但不应该真正影响使用该公司模块的大多数开发人员(这些模块是 RedisSearch、RedisGraph、RedisJSON、RedisML 和 RedisBloom)。

这肯定不是我们最后一次听说此事。迟早会有更多的项目走上同样的道路。到那时，我们可能会看到更多解决这个问题的标准许可证，这样其他公司就不必多次更换。尽管在理想情况下，我们不需要它，因为每个人都会表现得很好——但是因为我们不是生活在乌托邦里，这不太可能发生。
