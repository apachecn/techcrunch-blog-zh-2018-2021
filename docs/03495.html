亚马逊对开源的行为加上缺乏行业协会的领导，如开源倡议 (OSI)将扼杀开源创新，使商业开源更不可行。

结果将是更多的软件变得专有和封闭源代码，以保护自己免受AWS、广泛的许可证扩散(12家公司在2018年改变了他们的许可证)和开源许可证的影响，让位于一种新的许可证类别，称为源代码可用许可证。

不要误解我——仍然会有很多很多的开源软件。但是，如果我们作为一个行业不能解决亚马逊的问题，开源基础设施软件的作者将把他们有趣的功能放在他们的“企业”版本中。

不幸的是，我在11月写的地平线上的乌云越来越近了。亚马逊对开源展示了三种特别具有攻击性和侵略性的行为:

  • 它获取他人生产的开源代码，将其作为商业服务运行，并且不向生产和维护开源的商业实体提供任何回报，从而拦截开源的货币化。
  • 它分叉项目，强行从生产和维护开源项目的商业实体手中夺取控制权，就像Elasticsearch的情况一样。
  • 它劫持开源API并将其置于自己的专有解决方案之上，从而将客户从开源项目转移到自己的专有解决方案，就像它对MongoDB APIs所做的那样。

亚马逊对开源的行为是利己的，也是理性的。亚马逊正在遵守软件许可的规则。但是，如果行业协会创建标准的开源许可证，允许开源软件的作者表达一个简单的概念，这些行为及其不良后果就可以得到遏制:

“我不希望我的开源代码作为商业服务运行。”

领导力往往来自意想不到的来源。

但OSI，一个对许可证的开源性发表意见的组织，是一个无效的书呆子坦克，拒绝承认这个问题，并坚持认为，除非亚马逊有“自由”获取你的代码，将其作为商业服务运行，并且不给你任何回报，否则你的代码不是“开源的”。OSI认为它拥有开源的定义，并拒绝更新开源的定义，这是短视和危险的。

举例来说:服务器端公共许可证(SSPL)——由MongoDB率先提出的许可证提案——完全仿照Gnu通用公共许可证(GPL)和Affero通用公共许可证(AGPL)。SSPL是一个完全可服务的开源许可证，像GPL和AGPL一样，SSPL不是禁止软件作为服务运行，而是要求你开源所有你用来使软件作为服务可用的程序。

在SSPL被提议作为OSI的开源许可证候选人之后，一场长达数月的滑稽辩论随之而来，之后OSI正式发表了其预谋的意见，即SSPL不是开源许可证，尽管GPL和AGPL是开源的。目光短浅的OSI忘记了保持一致:如果SSPL不开源，那么GPL和AGPL也不应该开源。无论如何，MongoDB将继续使用SSPL，但它只是不会被称为“开源”，因为OSI说它拥有“开源”的定义，它不能被称为“开源”。太好了。

来源可用的许可证

亚马逊的行为和这种行业领导力的缺乏结合在一起，是否不可避免地会扼杀开源创新，使商业开源更不可行？我们是应该容忍更多的软件变得专有和闭源以保护自己免受AWS的攻击，还是容忍广泛的许可扩散？

我们已经看到了大量的许可证激增。MongoDB SSPL，汇合社区许可证(CCL)，时间尺度许可证(TSL)，Redis源代码可用许可证(RSAL)，Neo4J公共条款，蟑螂社区许可证(CCL)，Dgraph(现在使用蟑螂社区许可证)，Elastic许可证，Sourcegraph公平源代码许可证，MariaDB商业源代码许可证(BSL)…等等。

趋势是“源代码可用”许可，而不是“开源”许可，因为源代码可用许可没有被开源行业协会的短视所污染，不要求亚马逊有“自由”来获取你的代码，将其作为商业服务运行，并且不给你任何回报。

为此，由希瑟·米克(Heather Meeker)领导的一群开源律师将很快在T4开放一套“资源可用”许可证(T5)以征求社区意见，希瑟·米克是技术和开源法律领域受人尊敬且无可争议的领袖，他曾参与过T2公地条款和SSPL。

源代码可用许可证套件有望为开源软件的作者提供多种方法来应对来自云基础设施提供商的日益增长的威胁。该套件将提供简短的纯语言源代码可用的许可证；标准化最近采用的源可用许可证的模式；并允许用户和公司混合和匹配您想要施加的限制(例如，仅非商业使用，或仅增值，或无SaaS使用，或任何其他)。我相信这些框架将是开源的明智选择，因为OSI拒绝在解决Amazon问题上发挥领导作用。

AWS和反竞争行为

更广泛地说，大多数行业观察人士都清楚，AWS正在利用其市场支配力进行反竞争。除非有所改变，否则对亚马逊和AWS采取反垄断行动的呼声是不可避免的，即使AWS从亚马逊剥离。这个问题不仅仅是开源。

亚马逊对开源的行为是利己的，也是理性的。

在开源领域，如果亚马逊今天没有违反任何法律，那么防止或遏制其行为的许可证是至关重要的。而且，缺乏对“开源”一词的理解的开源行业协会的领导意味着开源许可证是遏制这种行为的最可行的解决方案。不一定要这样。

领导力往往来自意想不到的来源。有迹象表明，其他云基础设施提供商正在成为开源社区的真正盟友。以谷歌为例。2019年4月Google Cloud Next的主要公告引人注目，令人鼓舞。该公司宣布与Confluent、DataStax、Elastic、InfluxData、MongoDB、Neo4j和Redis Labs等受亚马逊行为影响最大的公司建立合作关系。

谷歌云的新任首席执行官托马斯·库里安的言论呼应了我去年所说的话。

TechCrunch的Frederic Lardinois写道:

谷歌正在采取一种与其竞争对手非常不同的开源方法，尤其是AWS。…“最重要的是，我们相信最终获胜的平台是那些能够促进而不是破坏生态系统的平台。我们真的从根本上相信，”(库里安)告诉我。“任何最终胜出的平台都是在培育而不是关闭一个生态系统。如果你看看开源公司，我们认为他们努力构建技术并使开发者能够使用它。”

谷歌与这些商业开源玩家结盟是明智的——AWS正在云战争中击败谷歌，在谷歌的云上给予最佳商业开源产品一流的地位将帮助谷歌赢得更多企业客户。

也许更重要的是，关于生态系统如何繁荣的立场和语言非常令人鼓舞。

披露:作者投资了许多受云基础设施提供商行为影响的开放公司，间接拥有亚马逊的股份，除了任何滥用开源或反竞争行为外，还是亚马逊的超级粉丝。
