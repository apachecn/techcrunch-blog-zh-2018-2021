苹果希望在 2019 年 WWDC 上为其更广泛的应用生态系统充电

原文：https://web.archive.org/web/https://techcrunch.com/2019/06/04/apple-looks-to-recharge-its-broader-app-ecosystem-at-wwdc-2019/

开发者工具往往不会成为技术生态系统之外的头条新闻。但苹果本周在全球开发者大会上宣布的开发者工具可能会对苹果众多平台(包括所有不是 iPhone 的平台，如 macOS、watchOS、tvOS 和现在的 iPadOS)上可供消费者使用的应用数量和质量产生重大影响。

在本周的活动间隙，开发者们特别谈论的一件事就是 SwiftUI。

五年前，随着 Swift 编程语言的推出，苹果开始让开发变得更加容易。本周在 WWDC，它发布了一个新的用户界面框架，名为 SwiftUI ，扩展了这一愿景。该框架是从头开始构建的，旨在帮助开发人员使用简单的声明性代码构建一个具有流畅动画的全功能用户界面。

对于开发人员来说，这意味着他们可以通过 SwiftUI 的自动功能节省大量时间，从而设计出设计良好且错误较少的应用程序。或者，正如苹果向开发者解释的那样，“不仅仅是更少的代码，而是更好的代码。”

它的简单性意味着消除可能突然出现的整类错误；它的代码易于阅读——就像有人向你解释用户界面；它让开发者可以跨平台重用更多的代码。

它还允许迭代变得更快。如果开发人员后来想改变他们应用程序用户界面的一部分，这是一个更快、更容易的改变。

SwiftUI 的框架有助于界面布局，使应用适应 iOS 13 的新黑暗模式，可访问性，从右到左的语言支持和国际化等。同样重要的是，SwiftUI 可以通过 iOS、iPadOS、macOS、watchOS 和 tvOS 中内置的相同 API 在苹果的应用生态系统中使用。

如果那些之前只专注于 iOS 的出版商在现有应用中采用新框架，这可能会推动他们进行跨平台开发。

他们将在多大程度上这样做取决于他们应用程序的具体情况，但 SwiftUI 将吸引新的开发人员以及希望首次入门的新手。

SwiftUI 新闻伴随着 Xcode 的新版本——Xcode 11——现在包括一个图形 UI 设计工具，允许开发人员通过 SwiftUI 构建用户界面，而不必编写代码。

当在可视化设计工具中对 UI 进行更改时，会自动生成 Swift 代码。然后，开发人员可以通过在 iPhone、iPad、iPod Touch、Apple Watch 和 Apple TV 等联网设备上运行应用程序，实时预览应用程序的外观和工作方式。

这使他们能够测试他们的代码在每个平台上的工作情况，包括应用程序如何响应多点触摸，或者在开发过程中如何与摄像头或其他传感器配合工作。

观看应用程序

对于 watchOS，SwiftUI 解决了为 Watch 应用程序构建动画和效果的复杂性问题——这限制了一些开发人员将 Watch 作为一个应用程序平台。

该框架将支持构建手表应用程序，具有滑动删除、重新排序列表项目、旋转滑动和直接访问数字表冠等功能。

这款手表还拥有自己的设备上应用商店和独立应用程序，即使没有 iOS 版也可以安装。(甚至是一部 iPhone。)

使用独立应用程序，开发人员可以将手表应用程序从 iOS 中分离出来，甚至可以将手表指定为独立的推送目标——这意味着他们只能向手表发送通知，而不是所有用户的平台。

手表应用还将能够支持 CloudKit 订阅和复杂推送，让用户了解最新动态。由于手表应用程序现在可以针对不使用 iPhone 版本应用程序的用户，他们也可以提供文本字段，用户可以输入用户名和密码，从他们的手腕上注册应用程序，或者他们可以使用苹果按钮登录，也是新宣布的(在某些情况下，甚至需要)。)

手表应用程序现在也可以传输音频，这为一种不同于以前的应用程序铺平了道路。不难想象——正如演示的那样——一个从基于互联网的流媒体服务(如 Pandora)播放直播体育或音乐的应用程序。

此外，watchOS 的新扩展运行时可能会推动另一种手表应用的开发，即使用户放下手腕，这种应用仍然可以运行。

例如，专注于自我护理、正念、物理治疗、智能警报或健康监测的应用程序可以利用这一点，为手表用户创造新的体验。

手表应用生态系统的最初版本速度较慢，不仅是因为构建应用的复杂性，还因为开发者受到的限制，不允许他们以某些方式针对手腕。开发者没有考虑什么在手腕上有意义——例如，使用传感器或流音频的应用——而是对现有的 iOS 应用进行了更多的基本移植。

不足为奇的是，那些中的许多失败了，后来被移除。这是苹果试图给手表应用生态系统第二次机会。

macOS 应用

对于 Mac，在 WWDC 宣布的新开发者工具将帮助 iOS 开发者接触到 Mac 的 1 亿活跃用户。

苹果表示，它意识到许多原生 iPad 应用程序在 Mac 上看起来很棒，但开发者没有时间使用 AppKit 来移植应用程序。因此，今年，它推出了一项技术，允许开发人员以“最小的”努力将 iPadOS 应用程序带到 Mac 上。

它谈到了目前超过 100 万个 iPad 应用的生态系统，其中许多应用在 Mac 上也有意义。

作为这项工作的一部分，苹果将 40 个框架从 iOS 移植到 Mac，几乎整个 iOS API 集都移植了，只有少数例外。苹果公司表示，这是通过将 UIKit 作为一个本地框架，并在新版本的 macOS 中直接集成到 macOS 中来实现的。

此外，苹果将苹果应用程序移植到苹果电脑分为三个步骤。

第一步是在 Xcode 的项目编辑器中选中一个写着“Mac”的框。

在 Xcode 中，当您更改源代码时，所有应用程序都会自动更新——包括在 iOS/IpaDo 和 Mac 上的应用程序。

开发人员被告知，他们首先需要一个支持最佳实践的出色的 iPad 应用程序，然后应该通过添加苹果特有的功能(如完整菜单、工具栏、悬停事件、触摸栏等)为苹果定制该应用程序。

这并不是制作一个伟大的苹果应用程序的复选框，但它要简单得多。

然而，问题仍然是苹果会在多大程度上执行“出色”的 iPad 应用要求。它表示，开发者应该采用 iPad 的最佳实践，比如支持外部键盘或者利用其他关键技术，比如 Metal，以获得最佳效果。

然而，如果苹果真的只是想用更多的应用程序来充实其 Mac 应用程序商店，以及更多的创收应用程序，它可能不会坚持这种水平的伟大 iPad 设计。

在 WWDC 之前，该公司已经与十几家开发商测试了这一过程，包括美国航空公司、Crew、DC Universe、Post It、Twitter、Tripit、Fender、Asphalt 9、吉拉等。

iPadOS

与此同时，随着 iPadOS 的推出，在 iPad 上运行的 iOS 也有了自己的品牌。

iPad 自发布以来一直由 iOS 驱动，但随着时间的推移，它开发了自己的一套专为更大的画布设计的特定功能，例如滑过、分割视图、拖放和对 Apple Pencil 的支持。

首先，iPadOS 将有一个更紧密的主屏幕图标网格，这意味着开发者的应用程序有更多的空间。应用程序小工具现在可以固定在主屏幕上，这是 iPad 应用程序占据空间和用户注意力的另一种方式。

但 iPad 的优势在于，它成为了笔记本电脑的替代品，有助于提高工作效率，也有助于创意艺术，比如素描和数字艺术。

对于生产力应用程序开发人员来说，iPadOS 弹出应用程序独立窗口的新功能——更像一台“真实”的电脑——将是有用的，另外还有 App Exposé的添加，以及三指复制、剪切、粘贴和撤销手势等新手势。

在开发人员专用工具方面，新的 PencilKit API 将允许第三方应用程序同样访问苹果新设计的铅笔工具。

然而，真正刺激更多 iPad 应用开发的可能是将 iPad 应用移植到 Mac 上的便利性。换句话说，开发人员可能会有动力充实他们的 iPad 应用程序，因为他们知道可以用比以前更少的工作量将工作复制到 Mac 上。

tvOS

鉴于对 SwiftUI 和将 iPad 应用移植到 Mac 的关注，苹果针对 Apple TV 的 tvOS 受到的关注较少——而且因为苹果刚刚举行了一场活动，其围绕 Apple TV 和其流媒体服务的雄心是一个关键焦点。

也就是说，SwiftUI 将在这里发挥作用，因为它将允许重用 tvOS 应用程序的代码。

AR & ML

除了通过简化过程来促进开发之外，苹果本周还推出了其他几项开发技术，包括其 ARKit (ARKit 3)的更新版本，这将允许更好的 AR 应用程序，允许动作捕捉，以及识别帧中人的能力，以便他们可以在 AR 对象的前后移动，等等。

苹果的 Core ML 3 允许开发者在他们的应用中构建、训练和部署机器学习，即使他们不是 ML 专家。

对关键技术的这些和其他改进，如 Metal 和 CreateML，将帮助这些领域的开发者构建更高质量的应用程序。

但是，苹果现在如何利用其最受欢迎的应用平台 iOS 来为其整个应用生态系统充电，这或许更令人兴奋和感兴趣。通过本周宣布的工具，苹果旨在精简和简化开发和设计，让更多人编写代码，并鼓励其应用程序开发者社区超越 iPhone 进行思考。
