你的安全团队可能是一个令人恼火的障碍——但不一定是这样的 TechCrunch

原文：https://web.archive.org/web/https://techcrunch.com/2019/08/07/your-security-team-is-probably-an-infuriating-obstacle-but-it-doesnt-have-to-be-this-way/

安全是空洞的、无意义的戏剧——或者，至少，这是大多数大公司的大多数员工学到的一课。安全性是您的密码每隔几个月就会过期，如果您是新员工或承包商，您将无法访问重要的服务，来自您从未见过的团队的一条令人不快的消息，解释您的新计划不被允许，当有人不想承认他们的真实原因时，这显然是一个方便的借口。安全是狗屁。

在我自己作为咨询公司首席技术官的职业生涯中，我可以举出比我想的还要多的例子。这家家喻户晓的公司的安全团队解释说云服务本来就不安全，直到有一天他们决定转向 AWS，并开始解释本地服务器本来就不安全。那些家喻户晓的公司向我们提供了大量关于服务器安全性的详细安全问卷，但他们的评估协议却无法理解我们的“嗯，所有东西都在 GitHub 和 GSuite 等的云中”。，我们没有自己的服务器”没有长达一小时的牵手通话的回应。

这就是为什么今天早上听到迪诺·戴·佐维在黑帽安全会议上的主题演讲是如此的令人振奋。Square 的安全工程师戴·佐维(Dai Zovi)认为，作为一个团队的安全模式太常见了，这种模式坐在那里对实际构建东西的人冷嘲热讽，告诉他们不要，指指点点，实际上适得其反。

相反，他认为，安全必须改变其文化，这比战略重要得多，而战略又比战术重要得多。团队应该对自己的安全负责，而不是让安全成为一个遥远的燃烧的铁环。此外，安全工程师应该编写代码来帮助这些团队。 Fuzzing 很棒，但正如他所说，“下一个阶段是让软件开发人员更容易使用 Fuzzing，因为他们的数量比我们的数量多得多。”

最重要的是——也是最具革命性的——他认为，安全人员不应该总是默认说“不”，并设置尽可能多的障碍，而应该总是从“是的，这就是我们如何能帮助你”开始事实上，这是如此不同于今天的做法，它实际上听起来滑稽说了很多，没有一个是好的。

可悲的事实是，今天，在企业软件的现实世界中，大多数员工和供应商遇到的安全问题往往至少像美国机场的“脱鞋&拿出你的液体”安全剧场一样毫无表演价值。恐怖故事数不胜数。你有你自己的，我肯定。谁没有呢？

再举几个例子:有一次，一家电影制片厂希望我们为与他们的知识产权没有真正联系的辅助网站做一些小的网络开发工作，他们告诉我们，除非我们的员工(主要是远程员工)能够连续使用门卡，并且能够通过和闭路摄像头监控可能在这些网站上工作的每台计算机……然后暗示说，他们真正需要的只是检查这些框，而不是让这些事情真正发生。

另一次，一家大公司坚持要求我们遵守 SOC-2——SOC-2 是一个标准，不是在技术领域而是在会计领域诞生的，似乎主要是为了给会计师提供充分就业，而不是有意义的安全标准和流程——而不关心我们正在谈论的 SOC-2 的五个“信任服务”中的哪一个(如果有的话);他们只需在供应商列表上勾选“SOC-2 兼容”即可。

不一定要这样。安全人员可以是贡献者，而不是看门人。如果是这样的话，每个人都会发现为安全做出贡献更容易、更有回报、也更直观。孤立的安全官僚机构不仅缓慢而且令人沮丧；从长远来看，与任何外部黑客甚至 APT 相比，它们对受其侵扰的公司的安全构成了更为根本的威胁。我们早就该吸取教训了。
