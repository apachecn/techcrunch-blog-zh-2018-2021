Mozilla在Firefox中加强了反跨站追踪，因为Chrome在隐私方面仍然落后

原文：https://web.archive.org/web/https://techcrunch.com/2021/02/24/mozilla-beefs-up-anti-cross-site-tracking-as-chrome-still-lags-on-privacy/amp/

图片来源: Getty Images

Mozilla在其Firefox浏览器中进一步加强了反跟踪措施。在昨天的一篇博客文章中，它宣布Firefox 86有一个额外的反cookie跟踪层，内置于增强跟踪保护(ETP)严格模式中——它称之为“总Cookie保护”(TCP)。

这一“重大隐私进步”，正如它所说，通过在每个网站上存储第三方cookies来防止跨网站跟踪。

Mozilla将这比作每个网站都有一个单独的cookie jar例如，脸书的cookie不会与你购买最新球鞋的运动鞋网站的cookie存储在同一个桶中，等等。

Mozilla解释说，新的隐私包装层“在Firefox的网站之间提供了cookies和其他网站数据的全面划分”。

除了上个月宣布的另一个反跟踪功能——针对所谓的“超级追踪器”——也就是偷偷摸摸的追踪器，将用户id存储在浏览器的“越来越模糊”的部分(如闪存、 ETags 和 HSTS 标志)，即用户很难删除或阻止它们的地方Mozilla称，这些功能结合起来“防止网站能够‘标记’你的浏览器，从而消除最普遍的跨站点跟踪技术”。

当跨站点cookies需要用于非跟踪目的时，有一个“有限的例外”——Mozilla给出了流行的第三方登录提供商的例子。

"只有当Total Cookie Protection检测到您打算使用某个提供商时，它才会允许该提供商专门为您当前访问的站点使用跨站点Cookie。这种短暂的例外允许强大的隐私保护，而不影响你的浏览体验，”它补充说。

长期以来，追踪器屏蔽一直是广告技术行业的一场军备竞赛，该行业决心继续监视网络用户，并对同意监视人们在线业务的想法嗤之以鼻，投入资源设计残忍的新技术，试图继续观察互联网用户在做什么。但近年来，随着浏览器制造商采取更强硬的支持隐私/反跟踪立场，这场斗争愈演愈烈。

例如，Mozilla在2018年开始让tracker阻止默认设置——接着在2019年让ETP成为Firefox的默认设置，阻止来自其合作伙伴Disconnect识别为跟踪器的公司的cookies。

而苹果的Safari浏览器在 2017 中增加了一项“智能跟踪预防”(ITP)功能——应用机器学习来识别跟踪器并隔离跨站脚本数据，以保护用户的浏览历史不被第三方发现。

谷歌还宣布计划逐步取消Chrome对第三方cookies的支持，这也让这只猫成为了广告技术的鸽子之一——它说这将在两年内于2020年1月在实现——尽管它仍在进行这个所谓的“隐私沙箱”项目(现在处于英国反垄断监管机构的监视之下)。

自2019年以来，谷歌一直在制造加强隐私的噪音，以回应浏览器市场其他人对在线隐私的担忧。

去年4月，谷歌撤销了一项让网站更难访问第三方cookies的改变，理由是担心网站在疫情期间能够执行基本功能——尽管这在7月份恢复了。但公平地说，在执行其声称的加强隐私的计划时，这家广告技术巨头仍然是落后者。

鉴于Chrome的市场份额，这使得世界上大多数网络用户比使用不同的、更注重隐私的浏览器暴露在更多的跟踪之下。

正如Mozilla最新的反cookie跟踪功能所示，战胜adtech对隐私(和同意)的过敏的竞赛也没有终点线。因此，在隐私保护方面行动迟缓与完全不提供隐私保护没有太大区别。

也就是说:一个令人担忧的发展——在非第三方cookie跟踪方面——在这篇由一组隐私研究人员撰写的新论文中有详细描述，他们对CNAME跟踪(也称为基于DNS的反跟踪规避技术)进行了分析，发现这种偷偷摸摸的反跟踪规避方法的使用在不到两年的时间里增加了约五分之一。

大约在 2019 年前后，当开发者发现法国一家报纸网站在野外使用该技术时，该技术一直在引发主流对“不可阻止”网络跟踪的担忧。研究显示，自那以后，使用量一直在上升。

简而言之，CNAME追踪技术通过将追踪器注入到被访问网站的第一方上下文中来隐藏追踪器——通过嵌入网站子域的内容，该子域实际上是追踪器域的别名。

“该方案的成功要归功于DNS授权。论文作者之一、隐私和安全研究员Lukasz Olejnik在关于这项研究的博客文章中写道:“最常见的是T4 DNS CNAME记录。追踪器在技术上被托管在被访问网站的子域中。

“采用这种计划会产生某些后果。这有点愚弄了基本的网络安全和隐私保护——认为用户是故意浏览追踪者网站。当网络浏览器看到这样的方案时，一些安全和隐私保护就放松了。”

不要被“放松”这个词的使用所迷惑——正如Olejnik继续强调的那样，CNAME跟踪技术“对网络安全和隐私有重大影响”。例如，浏览器被诱骗将追踪器视为被访问网站的合法第一方内容(这反过来会解锁“许多好处”，例如访问第一方cookies——这些cookie随后会被发送到由追踪器控制的远程第三方服务器，因此监控实体可以对个人数据进行恶意处理)。

因此，风险在于，通过屏蔽追踪器来保护隐私的一大部分巧妙的工程工作，可能会因为避开反追踪器的雷达而被搁置。

研究人员发现，一家(臭名昭著的)追踪器提供商Criteo在检测到Safari网络浏览器正在使用时，会将其追踪脚本恢复为定制的CNAME斗篷方案——据推测，这是一种绕过苹果ITP的方法。

对CNAME跟踪也有进一步的担忧:正如Olejnik所说，论文详细描述了作为当前网络架构的结果，该方案如何“打开了大范围cookie泄漏的途径”——解释了该技术如何被部署的结果是“许多不相关的、合法的cookie”被发送到tracker子域。

Olejnik在2014年的一项研究中记录了这种担忧，但他写道，这个问题现在已经爆发了:“作为冰山一角，我们在7377个网站上发现了广泛的数据泄露。几乎每个使用CNAME方案的网站都会发生一些数据泄露(分析cookies通常会泄露)。这表明这个计划非常危险。对网页安全和隐私有害。”

研究人员发现95%的研究网站上有cookies泄漏。

他们还报告说发现了其他第三方脚本设置的cookie的泄露，表明泄露的cookie在这些情况下会允许CNAME跟踪器跟踪网站上的用户。

在某些情况下，他们发现泄露的信息包含私人或敏感信息，如用户的全名、位置、电子邮件地址和(出于额外的安全考虑)身份验证cookie。

该论文继续提出了一些网络安全问题，例如当CNAME追踪器通过HTTP而不是HTTPS被提供服务时，他们发现这种情况经常发生，并可能为中间人攻击提供便利。

根据研究人员的说法，防御CNAME伪装方案将需要一些主要浏览器采用新的技巧——他们指出，虽然Firefox(全球市场份额约为4%)确实提供了对Chrome不提供的技术的防御。

支撑苹果Safari浏览器的WebKit引擎的工程师们也一直致力于增强ITP，旨在对抗CNAME追踪。

在去年11月的一篇博客文章中，IPT工程师John Wilander写道，作为对这种偷偷摸摸的技术的防御,“ITP现在检测第三方CNAME伪装请求，并将HTTP响应中设置的任何cookies的有效期限制为7天。这个上限与ITP对所有通过JavaScript创建的cookies的到期上限一致。”

勇敢的浏览器去年秋天也宣布了的变化，旨在打击CNAME伪装。

“在1.25.0版本中，uBlock Origin获得了使用Mozilla出色的浏览器. dns API 检测和阻止CNAME伪装请求的能力。然而，这个解决方案只能在Firefox中运行，因为Chromium不提供browser.dns API。在某种程度上，可以使用自定义DNS服务器阻止这些请求。然而，没有一款浏览器附带基于CNAME的广告拦截保护功能，并且默认开启，”它写道。

“在Brave 1.17中，Brave Shields现在将递归检查任何网络请求的规范名称记录，这些请求没有使用嵌入式DNS解析器进行阻止。如果请求具有CNAME记录，并且规范域下的相同请求将被阻塞，则该请求被阻塞。该解决方案默认开启，为数百万用户带来增强的隐私保护。”

但据研究人员称，市场份额最大的浏览器Chrome还有待改进，他们写道:

因为Chrome不支持用于扩展的DNS解析API，所以Firefox下的uBlock版]防御无法应用于该浏览器。因此，我们发现四个基于CNAME的追踪器(Oracle Eloqua、Eulerian、Criteo和Keyade)在Firefox上被uBlock Origin阻止，但在Chrome版本上没有。

在讨论研究结果时，论文的另一位作者汤姆·范·戈瑟姆(Tom Van Goethem)表示，该团队的数据显示，使用CNAME追踪的网络出版商正在使用它来补充其他更“典型”的第三方追踪方法。

“我们的数据显示，采用CNAME模式的出版商已经拥有相当数量的追踪器(平均每个网站20多个),并且发现这一数量随着时间的推移保持稳定。这表明，基于CNAME的跟踪并不是用来取代典型的第三方跟踪，而是用来增加他们的跟踪能力，例如，通过反跟踪机制来锁定用户，”他告诉TechCrunch。

研究人员并没有专注于为CNAME隐形应用的增加提供一个因果解释。但范·戈瑟姆说，他们发现一些追踪者只在Safari浏览器上使用它(“众所周知，Safari浏览器包括严格的反追踪保护”)——这表明(至少)部分解释了这种技术的使用越来越多。

当被问及他们希望看到什么样的社区反应来打击CNAME计划时，他说:“理想情况下，更多的反跟踪防御应该对CNAME的跟踪采取强有力的措施，这样无论用户使用什么浏览器，他们都受到保护。”

浏览器需要部署多重防御来对抗各种技术，才能有效捍卫用户隐私。因此，在Firefox的新TCP功能的情况下，Van Goethem指出，尽管它“本质上”不影响基于CNAME的(第一方)跟踪，但它确实防止使用第一方跟踪的网站之间的cookie同步。

“通过基于CNAME的跟踪，追踪器可以跟踪用户在一个特定网站(包括追踪器)上的活动，但它需要与其他网站同步Cookie，以将网站A上一个特定用户的跟踪活动与网站b上同一用户的活动联系起来。Total Cookie Protection可以防止这种情况，”他解释道。

其他非基于cookie的跟踪技术，如ike设备指纹识别和短暂指纹识别，也可以用来规避基于浏览器的对基于cookie的跟踪的防御。“这意味着防御需要完整，并提供针对所有类型的跟踪的保护，以保护用户，”per Van Goethem说。

“我们希望我们的大规模分析表明，基于CNAME的跟踪正在上升，这将提高人们的意识，并激励用户和浏览器供应商提高对这种跟踪机制的防御，”他补充说。

“此外，我们希望我们的研究将强调出版商因纳入基于CNAME的追踪器而面临的潜在安全影响。在我们的研究中，我们发现了两个影响出版商网站所有访问者的安全问题；在一个案例中，问题仍然没有解决(尽管多次尝试联系追踪者报告漏洞)，使数百个网站的访问者处于危险之中。”

他还表示，监管行动/执法也可能有助于遏制CNAME技术的使用。
