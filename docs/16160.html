拜登关于网络安全的行政命令应该包括行为透明

拜登政府今年春天宣布了一项行政命令，旨在加强政府网络安全防御，此前最近发生了几起重大黑客攻击事件，包括网络安全管理软件产品、微软 Exchange Server 和 Pulse Secure 事件，这些事件影响了许多联邦机构和私营公司。几周后，黑暗面勒索软件对 Colonial Pipeline 的攻击凸显了该命令的重要性。

网络行政命令的一个关键要素是“软件材料清单”(SBOM)，供应商将被要求提供作为联邦采购过程的一部分。SBOM 将详细说明给定产品中使用的确切软件组件，包括任何开源组件，使联邦机构更容易、更快地确定他们是否受到这些组件之一中发现的漏洞的影响。

SBOM 是加强联邦网络安全的重要一步，但这还不够。了解各种产品中包含的软件组件将有助于机构安全团队在漏洞暴露时更快地做出反应，但在其他情况下，如网络安全管理软件产品式的供应链攻击，偷偷插入软件组件，其影响是有限的。

在联邦层面建立软件产品披露标准将有利于私营部门的网络安全，并提高软件供应链的整体安全性。

这就是为什么拜登政府应该扩大网络行政命令，不仅包括 SBOM，还包括“行为透明”

透明度要求在技术上并不是一个新概念。证书透明度(CT)是由任何公共证书颁发机构(CA)颁发的所有证书的公共分类账，它提供了一个监控和审计 CA 活动的框架，而苹果最近宣布的应用程序跟踪透明度允许用户看到应用程序正在跟踪哪些活动，并选择退出。行为透明性是这一概念在已知软件行为中的应用。

行为透明框架的目的是列举给定软件将在设备或网络上采取的感兴趣的预期动作。这有助于安全分析师区分预期噪音和危害迹象。反过来，这可以让安全团队在识别任何专有或开源软件中未知漏洞的利用方面具有优势。

好消息是，列举常见的软件行为已经成为外部网络活动的标准行业惯例。大多数主流软件厂商，包括 Meraki 、 McAfee 、 Tenable 、 LogMeIn/GoToMeeting ，以及我自己的公司 ExtraHop ，都已经发布了常见产品行为列表。甚至网络安全管理软件产品也有描述其网络行为的文档。

但拜登政府可以帮助实现重大变革，改善这一行业实践，并改善公共和私人组织的整体安全态势。

建立行为透明的标准

首先，网络行政命令应与有代表性的软件和安全软件供应商以及 MITRE 等组织合作成立一个工作组，为实现完全行为透明必须包括的网络活动类型创建标准。

至少应包括外部网络目标、与其他软件组件的内部网络连接行为，以及相关网络端口列表(如果适用)和这些端口的用途。行为透明框架还应该包括其他网络行为，特别是(但不限于)任何看起来像扫描或侦察行为的行为。

让通用安全工具可以使用行为数据

其次，网络行政命令应要求以 JSON 或 CSV 等机器可读格式发布已知的软件行为，这些格式可以被安全信息和事件管理(SIEM)、防火墙、端点保护平台、网络检测和响应以及变更管理工具等常见安全产品吸收。

这是与当前模型的一个重要区别，在当前模型中，大多数行为都列在网页上或机器可读的 PDF 文件中。有了这种变化，通用安全工具可以使用机器可读的行为数据来帮助建立组织内活动的基线，以便更快、更准确地检测到指示危害的偏差。Meraki 已经通过提供 CSV 格式的列表做到了这一点。

集中访问行为信息

第三，网络行政命令应该建立一个行为透明度数据交换所，由网络安全和基础设施安全局或其他适当的联邦机构管理。现状是在一个供应商的网站上搜寻，查阅他们的产品文档或打开一个支持案例来找出网络行为。如果提供的信息不正确，这也是一个支持案例。

目前分散的方法有很大的问题。企业软件产品不受约束的网络访问带来了巨大的安全风险——零信任框架正是为了防止这种风险而建立的——但典型的从业者没有时间或专业知识来单独跟踪环境中每个企业软件的预期行为。如果没有对行为透明性数据的集中访问，即使是最好的零信任实现也会在企业软件方面存在重大差距。

信息交换机构将为行为透明度数据提供一个集中的存储库，按公司、产品和产品版本进行组织。像 GitHub 这样的论坛是这样一个交换中心的理想机制，为这些信息提供了一个广泛使用的集中存储库。

简化用户和供应商之间的反馈

第四，交换所应该包括一种机制，通过这种机制，产品用户可以很容易地向软件供应商提供反馈。反馈可以是问题的形式，甚至是拉式请求，尽管公司应该参与批准变更。这样，可以在公共论坛上指出行为中的缺陷。大多数缺陷的原因是产品更新没有反映在行为透明度数据中，尽管随着时间的推移，公司理想的做法是确保这些数据保持最新。但是也会发现真正的积极因素。

用行为透明保护软件供应链

2020 年 12 月首次披露的网络安全管理软件产品软件供应链攻击说明并强调了行为透明的重要性。在 12 月 11 日之前，当 FireEye 首次发现网络安全管理软件产品猎户座软件中的漏洞时，至少有两家其他网络安全公司，帕洛阿尔托和费德利斯发现，他们在网络安全管理软件产品的安装与攻击者控制的“阶段 1”avsvmcloud[通信。]com 域。Palo Alto 观察并阻止了其他恶意行为，但当时两家公司都没有确定与 avsvmcloud[。]com 本身就受到怀疑。这在很大程度上是因为查看网络数据时存在大量众所周知的“噪音”。

但是，如果更多的组织能够获得网络安全管理软件产品的行为透明度数据，以及一个比较偏离基线的论坛，事情可能会有所不同。

网络安全管理软件产品猎户座并没有触及很多外部目的地，所以当供应链攻击的第一阶段开始触及“app sync-API . eu-west-1 . avsvmcloud[。]com，“运行 SIEM 查询的威胁搜索分析师，或基于机器学习的 EDR 或 NDR 产品，可能会更快地确定出了问题。

同样，一个低摩擦的公众反馈机制可能已经向网络安全管理软件产品和行业暗示，孤立地看起来像噪音的东西(“appsync-api，似乎是合法的？”)实际上是更邪恶的东西。

网络行政命令以及对俄罗斯的制裁是强有力的早期迹象，表明拜登政府打算采取更加积极主动的方法来应对网络安全。这些努力成功的关键将是政府与私营部门技术提供商建立的伙伴关系。在联邦层面建立软件产品披露标准将有利于私营部门的网络安全，并提高软件供应链的整体安全性。
