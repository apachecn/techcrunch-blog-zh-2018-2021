我在谷歌工作了六年。在内部，你别无选择——如果你正在部署微服务和容器，你必须使用 Kubernetes(在 Google 内部实际上不叫 Kubernetes；叫博格)。但是这个曾经只是谷歌内部项目的项目已经被开源，并成为软件开发和运营中最受关注的技术之一。

有充分的理由。一个人用一台笔记本电脑就可以完成过去需要一大群工程师才能完成的工作。有时候，Kubernetes 感觉像一个超级大国，但是伴随着可伸缩性和敏捷性的所有好处而来的是巨大的复杂性。事实是，很少有软件开发人员真正了解 Kubernetes 是如何工作的。

我喜欢用手表来比喻。从用户的角度来说，它是非常简单的，直到它坏掉。要真正修理一只坏了的手表，需要大多数人都不具备的专业技能——我向你保证，Kubernetes 比你的手表复杂得多。

大多数团队是如何解决这个问题的？事实是，他们中的许多人不是。他们经常采用 Kubernetes 作为他们数字化转型的一部分，却发现这比他们预期的要复杂得多。然后他们不得不雇用更多的工程师和专家来管理它，这在某种程度上违背了它的目的。

在你看到容器的地方，你会看到 Kubernetes 来帮助编排。根据 Datadog 关于容器采用的最新报告，将近 90%的容器都是经过编排的。

所有这些都意味着 DevOps 初创公司有一个很好的机会来解决 Kubernetes 生态系统中的不同痛点。这项技术不会有任何发展，所以任何有助于使它更加安全、易于使用和易于故障排除的平台或工具都会受到软件开发社区的欢迎。

从这个意义上说，风投投资这个生态系统的时机再好不过了。我相信 Kubernetes 正在成为新的 Linux:全球 100 万台网络服务器中 96.4%的操作系统都是 Linux。类似地，Kubernetes 正在成为现代云原生应用的事实上的操作系统。它已经是云原生计算基金会(CNCF)内最受欢迎的开源项目，91%的受访者使用它-从 2019 年的 78%和 2018 年的 58%稳步增长。

虽然这项技术已经过验证，采用率也在飙升，但仍然存在一些基本挑战，这些挑战无疑将由第三方解决方案来解决。让我们更深入地看看为什么我们会在这个领域看到创业公司激增的五个原因。

 

容器是构建现代应用程序的首选方法

Docker 彻底改变了开发人员构建和发布应用程序的方式。容器技术使得在云之间移动应用程序和工作负载变得更加容易。它还提供了与传统虚拟机管理程序一样多的资源隔离，但有相当多的机会来提高敏捷性、效率和速度。
